# Org Canvas Chat

The agent chat that lives in the right panel of the org canvas (the **Chat** tab of `OrgRightPanel`, rendered by `SidebarChat`). This doc covers the chat subsystem: the Zustand store, autosave + live-sync persistence, the send/stream path, the planner fan-out → sub-agent cards, the attention-list intro, and title generation.

**Sibling doc:** `CANVAS.md` — the canvas itself (projection pipeline, categories/geometry, `+` menu, drag/drop, edges/connections, deep links, right-panel layout). Read it for anything that isn't the chat. The two are intentionally intertwined (the store exists *for* bidirectional chat ↔ canvas flow), so cross-references are expected.

The agent-proposal flow (`<ProposalCard>` rendered inline in chat, Approve/Reject via `approval`/`rejection` fields on user messages) is documented in `CANVAS.md` under "Agent-proposal flow" — it creates canvas entities, so it lives with the canvas doc. The chat-side consequence of approving a *feature* proposal (the planner fan-out → `<SubAgentRunCard>`) is documented here.

## The short version

- `src/app/org/[githubLogin]/_state/` — Zustand store + side-effect hooks for the canvas chat. `canvasChatStore.ts` is the source of truth for conversations, messages, streaming state, and reserved slots (`proposals`, `subAgentRuns`, `artifacts`) that future canvas-bound features will fill. `useSendCanvasChatMessage.ts` owns the streaming send path (fetch → `useStreamProcessor` → store writes). `useCanvasChatAutoSave.ts` is the write-through subscription that persists messages to `chat_conversations` via the org endpoint, AND the live-sync that merges server-appended rows back in. **Persistence is identity-based, NOT count-based** — `persistedIdsRef` tracks which message *ids* are on the server; the save delta is "local ids not yet persisted" and live-sync MERGES server rows by id (never wholesale-replaces). The pure invariant helpers live in `canvasChatPersistence.ts` (`seedPersistedIds` / `computeUnsaved` / `mergeServerMessages`) so the "never lose a message" contract is unit-testable without the store. This replaced an older count + seed-offset scheme that could shift the save window by one and silently drop the user's first message when a Pusher nudge interleaved with a PUT. The store is **org-canvas-scoped on purpose** — the dashboard chat keeps its own local state and does NOT use this store.
- **Org chat → agent toolset wiring (capability registry).** `src/app/api/ask/quick/route.ts` is the single chat stream endpoint; when `orgId` is supplied (the canvas chat, not the dashboard chat) it calls `runCanvasAgent`/`ln` after a `validateUserBelongsToOrg` check. The org tools + prompt suffix + readonly strip set are no longer merged ad-hoc — they're composed from `CAPABILITY_REGISTRY` in `src/lib/ai/capabilities.ts`. Each **capability** bundles a `buildTools` factory, a prompt snippet, its write-tool names, and a `core` flag. The five: `roadmap` + `planner` (CORE, always in the prompt — the hot path: propose a feature, then drive it with `send_to_feature_planner`), and `whiteboard` + `research` + `connections` (LOADABLE — **progressive disclosure**). Loadable snippets are NOT in the up-front prompt; the core suffix carries a one-line menu and the agent calls the `learn_capability` tool to pull a snippet on demand (the tools themselves are always registered since the AI SDK fixes the toolset at call start). This keeps the always-on prompt ~8k tokens instead of ~13k. Prompt snippets live in `src/lib/constants/prompt.ts` (`getRoadmapCapabilitySnippet`, `getWhiteboardCapabilitySnippet`, `getPlannerCapabilitySnippet`, `getResearchCapabilitySnippet`, `getConnectionsCapabilitySnippet`; `getCanvasPromptSuffix` is the back-compat full inline composition). Adding a capability = new entry in `CAPABILITY_REGISTRY` (factory + snippet + `core` flag + `menuBlurb` if loadable); `composeCapabilityTools` / `composeCapabilityPromptSuffix` / `composeWriteToolNames` pick it up automatically.

## Gotchas

- **Canvas chat state lives in `useCanvasChatStore` (`_state/canvasChatStore.ts`), not in `SidebarChat` local state.** The store exists because several upcoming features — LLM-generated `propose-canvas-change` artifacts (rendered as both a chat card AND a halo on the affected canvas node), sub-agents (kicked off from chat, surfaced on the nodes they're working on), live task-status / PR-list artifacts — create bidirectional flow between the chat and the canvas itself. Two `useState` islands talking via prop drilling would be miserable. The store is the single source of truth; chat and canvas both subscribe with narrow selectors. **Conversation lifecycle is owned by `OrgCanvasView`, not `SidebarChat`** — `OrgCanvasView` calls `startConversation` once `chatReady` is true (workspaces + hidden list + share preload all resolved), then keeps the conversation's `context` (canvas ref, breadcrumb, selected node id) in sync via `updateActiveContext`. `SidebarChat` is a pure renderer over the active conversation; it never owns chat state, so unmounting it (e.g. a tab switch) is cheap and idempotent. **Auto-save runs at the page level**, not inside `SidebarChat` — `useCanvasChatAutoSave` mounts in `OrgCanvasView` and survives chat unmounts, so pending writes never get dropped by tab navigation.
- **Store performance contract — read this before adding a consumer.** The streaming write path commits one `set()` per stream chunk (~50/sec). To keep that cheap:
  1. **Always select.** Never call `useCanvasChatStore()` without a selector — that subscribes to the whole store and re-renders on every text-delta. Use `useCanvasChatStore((s) => s.x)` instead. The single biggest footgun.
  2. **Use `useShallow` for derived collections.** A selector that returns a fresh array/object on every call (e.g. `Array.from(s.conversations.values())` or `messages.map(...)`) defeats Zustand's `Object.is` bail-out → re-renders constantly. Wrap with `useShallow` from `zustand/react/shallow`.
  3. **Stable empty-array sentinels.** When a selector might return `undefined` (e.g. no active conversation), fall back to a module-level `const EMPTY_X: T[] = []` instead of `?? []` — the latter creates a new `[]` every time, breaking identity equality. See `EMPTY_MESSAGES` / `EMPTY_TOOL_CALLS` in `SidebarChat.tsx`.
  4. **Select narrowly.** `SidebarChat` selects `messages`, `isLoading`, `activeToolCalls` separately rather than the whole conversation object, so the canvas's `proposals` selector is unaffected by chat streaming, and vice versa. When the canvas grows badge/halo subscribers for proposals/sub-agents/artifacts, follow the same pattern: select the specific slice you render, not "the conversation."
- **Identity-based autosave — never lose a message.** `useCanvasChatAutoSave` tracks a per-conversation `persistedIdsRef: Set<messageId>`, not a count. The save delta is `computeUnsaved` (local messages whose id isn't persisted, in order — so a creating POST always *leads with the real first user message*, never a placeholder-titled assistant row), and live-sync uses `mergeServerMessages` (keep all local rows, append server rows we don't already have, by id). Both operate on identity, so a mid-turn Pusher nudge can never drop a local message: merge only adds; the delta only sends ids not yet on the server. The merge runs only when the conversation is idle (no unsaved local messages, not streaming, no save in flight); a nudge that arrives mid-turn is deferred and retried after the next successful save. Ephemeral seeds (the AttentionList intro; the messages a joined/share conversation already has on the server) are pre-seeded into `persistedIdsRef` so they're never re-sent. The pure helpers in `canvasChatPersistence.ts` are unit-tested in `src/__tests__/unit/canvas/canvasChatPersistence.test.ts` (incl. the seed→user→planner-nudge lifecycle that used to drop the first message).
- **Conversation title self-heals.** The title is generated once at create time from the first user message (`generateTitle` in `src/lib/ai/conversationHelpers.ts`). If the creating POST's delta ever leads with a non-user row, the title freezes as `UNTITLED_CONVERSATION`; the org PUT route recomputes it from the full message list the moment a user message lands (and the history popover treats the placeholder as empty, falling back to the first-user-message preview). The identity-based autosave (above) makes the freeze rare — the creating delta now always leads with the real user message — but the heal covers legacy rows.
- **Approved feature → `<SubAgentRunCard>` requires the conversation id to round-trip.** When the user approves a feature proposal, `handleApproval.approveFeature` stamps `Feature.parentCanvasConversationId = <SharedConversation.id>` and seeds the feature's plan chat → the Stakwork planner runs → its plan posts back through the `/api/chat/response` webhook → `fanOutPlannerMessageToCanvas` appends a `source.kind === "planner"` row into THAT conversation → Pusher nudge → `useCanvasChatAutoSave` live-sync merges it → `getSubAgentRunsFromMessages` renders the card. The whole chain hinges on the stamp, which needs the **server** conversation id at approval time: (1) the client (`useSendCanvasChatMessage`) sends `conversationId: conv.serverConversationId` in the `/api/ask/quick` body, and (2) the route validates it with `resolveOrgConversationRowId` (org-scoped) — **NOT** `resolveTokenAttributionRowId`, which is workspace-keyed and silently returns null for org-canvas rows (`workspaceId: null`), leaving the feature un-fanned-out and the card permanently absent. If you add a new approval side-effect that fans out to the conversation, keep both halves wired. (Diagnose a missing card with the read-only `scripts/diagnose-subagent-card.ts`: it reports, per recent feature, whether the stamp landed, and per conversation, whether a planner row is present — disambiguating the data layer from the render layer below.)
- **`<SubAgentRunCard>` renders under SUPPRESSED planner / form-answer rows.** The card hangs under the message whose id == `run.anchorMessageId`. For an **inbound-only run** — the approval flow, where the canvas agent never makes an outbound `send_to_feature_planner` tool call, so the planner's reply is the only activity and therefore the anchor — that anchor is a `source.kind === "planner"` message, whose *bubble* `SidebarChat` suppresses (`return null`-style). So `SidebarChat` must render the bubble-suppressed planner / `user-answered-planner-form` rows' anchored card anyway (via the shared `renderSubAgentRuns` helper), or the card vanishes the instant the planner replies. **Do not** restore a plain `return null` for those rows — that's the exact bug where an approved feature's clarifying-questions card never showed. The card moving "down with the latest exchange" is intentional: an inbound reply moves the anchor, which is why it must be renderable at a suppressed-row position.
- **Planner-source rows carry feature display metadata + describe content-less replies.** Inbound-only runs have no outbound tool-call output to seed the feature name/workspace from, so the fan-out echoes `featureTitle` / `workspaceSlug` / `workspaceName` onto the planner row's `source` (the `/api/chat/response` webhook selects them from the `Feature`), and `getSubAgentRunsFromMessages` seeds the run from there — without it the card reads "Unknown feature" with no plan link. All three fields are optional (legacy rows fall back to the placeholder). Separately, planner replies often carry **no prose** — the payload is a clarifying-questions FORM or a TASKS-breakdown artifact, not message text — so `runMessageDisplayText` renders a descriptive label ("Asked a clarifying question" / "Generated a task breakdown" / "Posted the plan" / "Posted an update") instead of a bare "(empty)". Outbound content-less sends keep the literal "(empty)".
- **Adding a canvas-bound chat feature?** Use the reserved slots in `useCanvasChatStore`. Proposals → `state.proposals` map; long-running agent runs → `state.subAgentRuns`; rich artifacts → `state.artifacts`. Each is keyed by id; messages reference them via `message.artifactIds`. The chat renders the inline card from `state.artifacts[id]`; the canvas reads the same id from the same map and renders its badge. If you find yourself emitting a Pusher event from the chat to the canvas (or vice versa), stop — that's what the store is for.
- **Synthetic intro / "top items needing your attention".** First registered artifact type is `attention-list` (`src/app/org/[githubLogin]/_components/AttentionList.tsx`). On fresh canvas entry `OrgCanvasView` fetches `/api/orgs/[githubLogin]/attention?limit=3` (backed by `src/services/attention/topItems.ts`, which fans 4 signal queries — halted/failed workflows, features awaiting user feedback, tasks with FORM artifact, ready-to-review COMPLETED — across the user's accessible workspaces in the org), then seeds `startConversation` with one synthetic assistant message + a registered artifact of `type: "attention-list"`. The seed is **ephemeral**: `startConversation` accepts `ephemeralSeedCount`, which `useCanvasChatAutoSave` reads to skip leading messages from the first POST/PUT — so the synthetic intro never lands in `chat_conversations` and never leaks into `?chat=<shareId>` shares. Skip conditions: `?chat=<shareId>` is present (forking), or `sessionStorage["hive:attention-dismissed:<login>"] === "1"` (user clicked × this session). Dismissal flows through `state.dismissedArtifactIds` (a generic per-session dismiss set the dispatcher reads) — `OrgCanvasView` subscribes once and writes the sessionStorage flag whenever an `attention-*` id lands in it. **`MessageArtifacts` in `SidebarChat` is the single dispatcher** — switch on `artifact.type` and add a render arm. Per-card behavior (callbacks, etc.) goes through store actions, not function-valued artifact data, so the artifact registry stays serializable.
- **Chat input prefill via `pendingInputDraft`.** A canvas affordance can compose a chat message *for* the user without a DB write or agent tool call: write the text to `useCanvasChatStore.pendingInputDraft` and switch the panel to the Chat tab. `<SidebarChatInput>` watches the slot in an effect, applies the value to its local `input` state, focuses + caret-to-end, then immediately clears the slot via `setPendingInputDraft(null)` so a tab-switch back doesn't re-apply. The user can edit and send normally. The canonical use today is the Connections link-mode `+ Create connection` button (`Make a connection document between A and B`). This is the canonical pattern for "canvas affordance composes a chat message" — proposals, agent suggestions, etc. should reuse `pendingInputDraft` rather than reaching imperatively into the textarea ref.
- **What's NOT in this store.** Voice transcript / mic state lives in `src/stores/useVoiceStore.ts` (separate concern, separate lifetime — mic-session-scoped, available across surfaces). When voice transcription wants to drop a finalized transcript into the canvas chat, it calls `useCanvasChatStore.getState().appendUserMessage(...)`. **The dashboard chat does NOT use this store.** `DashboardChat` keeps its own local state on purpose; the two surfaces are diverging. Don't try to unify them — when a feature lives on both, share the lower-level pieces (`useStreamProcessor`, message types), not the state layer.
