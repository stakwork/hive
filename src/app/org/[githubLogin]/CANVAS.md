# Org Canvas

A live whiteboard on the org page. Initiatives and Milestones are real DB rows projected onto the canvas; humans create them through the `+` menu (which opens a dialog and hits the REST API). The agent annotates around them.

**Design doc:** `docs/plans/org-initiatives.md` — read it before non-trivial changes. (Old plan `docs/plans/org-canvas.md` is superseded.)

## The short version

- `src/lib/canvas/` — projection pipeline. `readCanvas(orgId, ref)` merges DB-projected live nodes (workspaces, repos, initiatives, milestones) with the authored `Canvas.data` blob. `writeCanvas` splits merged docs back into an authored-only blob. `geometry.ts` is the shared source of truth for card sizes / row layout. `pusher.ts` is the shared `CANVAS_UPDATED` emitter.
- `src/lib/ai/canvasTools.ts` — agent tools (`read_canvas`, `update_canvas`, `patch_canvas`), each with optional `ref`. Agent does NOT create initiatives or milestones — those are human-only via the `+` menu. Edge `customData` round-trips on read AND write (`compactEdge` surfaces it; `edgeInputSchema` accepts it; `update_edge` shallow-merges it like `update_node` does) so `customData.connectionId` (the slug pointing at a Connection doc — see "Edges link to Connection docs" below) survives every agent mutation. The system-canvas `CanvasEdge` TS type doesn't formally declare `customData`, but the splitter (`io.ts`) passes the field through untouched, so the cast-through-`as` pattern in `compactEdge` / `toCanvasEdge` is the same one `OrgCanvasView.readEdgeConnectionId` uses; don't try to "fix" it by adding the field to the lib's type — it's an intentional informal extension.
- `src/lib/ai/initiativeTools.ts` — agent tools for the DB-backed initiative ↔ milestone ↔ feature hierarchy. Five tools today: `read_initiative` and `read_milestone` (full-detail reads by id — `read_canvas` only returns the projector's render-time shape, not `description`/`assignee`/dates/etc.; both delegate to `loadNodeDetail` in `@/services/orgs/nodeDetail` so the cross-org guard stays in lock-step with the right-panel REST route); `assign_feature_to_initiative` (organize existing features under existing initiatives/milestones — DB-write); `propose_initiative` and `propose_feature` (suggest new ones, NO DB-write — see proposal flow below). All validate org ownership; mutations reuse `notifyFeatureReassignmentRefresh` so every affected canvas (root, both initiatives, both milestones, the workspace) gets a `CANVAS_UPDATED` fan-out. **Names beat ids for any user-facing UI**: when a propose tool validates a workspace / initiative / milestone, it pulls `name` in the same query and stuffs it into a render-only `meta` block on the `ProposalOutput` (`FeatureProposalMeta`, `MilestoneProposalMeta` in `src/lib/proposals/types.ts`). The proposal card reads from `proposal.meta` first and falls back to a cuid-suffix hint only for old (pre-meta) proposals. New propose tools or new card-rendered ids must follow the same pattern — never display a cuid in the card.
- **Agent placement vocabulary.** All three `propose_*` tools accept a `placement` field (`auto | near:<liveId> | above:<liveId> | below:<liveId> | left-of:<liveId> | right-of:<liveId>`). The agent never emits raw pixels — it picks a verb plus a live anchor id it saw in `read_canvas` output. Resolution lives in `src/lib/canvas/placement.ts:resolvePlacement`, which reads the target canvas, finds the anchor by id, and computes coords using `geometry.ts` dimensions (single source of truth shared with the projector). **Forgiving on failure**: malformed verb, anchor missing on the target canvas, or candidate slot collides with an existing card → return `null` and log a warning; caller falls back to projector auto-layout. We never fail an approval over a layout hint — LLMs hallucinate ids. `near` is an alias of `right-of` (deterministic direction; "find nearest free slot" can be a later enrichment without changing the agent surface). Resolution priority in `handleApproval` (per kind): (1) agent `placement` resolves cleanly → use it; (2) else, user's click hint (`intent.viewport`) on the matching canvas → use it; (3) else, no overlay → projector decides. Adding a new verb = extend `Placement` type in `src/lib/proposals/types.ts` + `computeCandidate` switch in `placement.ts` + the prompt vocabulary in `getCanvasPromptSuffix`.
- `src/lib/ai/researchTools.ts` and `src/lib/ai/connectionTools.ts` — agent tools for the two org-scoped doc types. Each has a four-tool surface today: `save_*` and `update_*` (the original create/update pair) plus `list_*` (compact enumeration with presence flags / status) and `read_*` (full body by slug). The read tools matter because both `update_*` paths are full-replace (research) or per-field-overwrite (connection); to extend rather than clobber, the agent must `read_*` first and send back combined content. The agent can also discover Connection slugs from `read_canvas` output via `edge.customData.connectionId` — that's the only place the slug↔edge linkage lives (no schema column on Connection for endpoints).
- `src/services/orgs/nodeDetail.ts` — shared `loadNodeDetail(kind, id, orgId)` helper. Single source of truth for the live-node detail shape (`{ kind, id, name, description, extras }`) and the cross-org guard. Two consumers today: the `node/[liveId]` REST route (right-panel Details tab) and the `read_initiative` / `read_milestone` agent tools. Add a new live-kind detail = extend the switch here; both consumers pick it up automatically.
- **Agent-proposal flow** (`src/lib/proposals/`, `docs/plans/agent-proposed-initiatives-features.md`, `docs/plans/propose-milestone.md`). The agent can `propose_initiative` / `propose_feature` / `propose_milestone` and the user clicks ✓/✗ on a `<ProposalCard>` rendered inline in chat; only on Approve does a DB row get created. Milestone proposals are the only kind that carry a side-effect list (`featureIds`) — approval creates the milestone *and* PATCHes those features' `milestoneId` in one transaction, then `notifyFeatureReassignmentRefresh` fires per-feature so synthetic feature→milestone edges appear via the next projector run with no extra client work. **The chat IS the source of truth for proposal lifecycle** — proposals are tool-call outputs in `CanvasChatMessage.toolCalls[].output`; approval/rejection ride as structured `approval`/`rejection` fields on user messages; the synthetic assistant message after approval carries `approvalResult` with `{ proposalId, kind, createdEntityId, landedOn }`. Status is *derived* by scanning the conversation (`getProposalStatus` in `src/lib/proposals/types.ts`) — no DB tables, no schema migration, no rehydration logic. Forks inherit the proposal trail naturally. The approval handler lives as a pre-LLM block in `/api/ask/quick`: it detects `approvalIntent`/`rejectionIntent` on the latest user message, runs `handleApproval` (validate org ownership, idempotency-scan via existing `approvalResult` messages, resolve `parentProposalId`, create the row, write a `Canvas.data.positions[liveId]` overlay only when `featureProjectsOn(currentRef, payload)` agrees, fan out `CANVAS_UPDATED`), and returns a synthetic SSE stream + an `X-Approval-Result` HTTP header that the client reads off the response and stamps onto the assistant message before autosave persists. Idempotency is naturally bounded by the conversation's append-only nature: re-clicking Approve appends another approval message, the scan finds the existing `approvalResult`, returns the same `createdEntityId`. **Single source of truth for the "feature lands here vs there" rule:** `featureProjectsOn` / `mostSpecificRef` in `src/lib/canvas/feature-projection.ts` — the projector and the approval handler both call it, drift is impossible.
- **Org chat → agent toolset wiring.** `src/app/api/ask/quick/route.ts` is the single chat stream endpoint. When `orgId` is supplied (i.e. the canvas chat, not the dashboard chat), it merges `buildConnectionTools` + `buildCanvasTools` + `buildInitiativeTools` on top of the per-workspace `askToolsMulti` toolset — after a `validateUserBelongsToOrg` check. The prompt (`getCanvasPromptSuffix`) teaches the agent to pick between them by intent (document-an-integration vs draw-a-diagram vs file-features-under-initiative). Adding a new org-scoped agent capability = new `buildXTools(orgId, ...)` factory + spread it into that same merge block + extend the prompt suffix.
- `src/app/api/orgs/[githubLogin]/canvas/*` — REST routes (root + `[ref]` + `hide` + `node/[liveId]`), all thin wrappers over `@/lib/canvas`. The `node/[liveId]` route is the right-panel's live-node detail endpoint; it now delegates to `loadNodeDetail` in `@/services/orgs/nodeDetail` so the agent tools `read_initiative` / `read_milestone` share the exact same cross-org guard and response shape.
- `src/app/api/orgs/[githubLogin]/initiatives/*` — REST routes for the DB-backed Initiative + Milestone models. Every mutating call emits `CANVAS_UPDATED` so open canvases refetch.
- `src/app/org/[githubLogin]/layout.tsx` + `_components/OrgShell.tsx` — server layout does auth + org lookup once; `OrgShell` renders the left icon rail (`OrgRail`) and either full-bleed children (canvas / schematic / graph) or a centered max-w container with org header (initiatives / workspaces / members). `useOrgView` maps pathname → view id and `viewIsFullBleed` decides which layout mode.
- `src/app/org/[githubLogin]/page.tsx` + `_components/OrgCanvasView.tsx` — default route. `OrgCanvasView` is the two-layer composition: full-bleed canvas (`OrgCanvasBackground`) on the left, fixed-width right panel (`OrgRightPanel`) on the right with three tabs — **Chat** (`SidebarChat`, the default landing tab; `_components/SidebarChat.tsx`), **Details**, and **Connections**. The canvas chat is also forkably shareable: the Share button in the chat header copies a `/org/<login>?chat=<shareId>` URL; landing on that URL preloads the conversation into a fresh continuable chat (each forker's own auto-save row).
- `src/app/org/[githubLogin]/_state/` — Zustand store + side-effect hooks for the canvas chat. `canvasChatStore.ts` is the source of truth for conversations, messages, streaming state, and reserved slots (`proposals`, `subAgentRuns`, `artifacts`) that future canvas-bound features will fill. `useSendCanvasChatMessage.ts` owns the streaming send path (fetch → `useStreamProcessor` → store writes). `useCanvasChatAutoSave.ts` is the write-through subscription that persists messages to `chat_conversations` via the workspace endpoint. The store is **org-canvas-scoped on purpose** — the dashboard chat keeps its own local state and does NOT use this store.
- `src/app/org/[githubLogin]/connections/` — canvas internals: `OrgCanvasBackground.tsx` (the canvas UI itself), `canvas-categories.ts` (category registry + scope rules — single source of truth), `canvas-theme.ts` (renderer definitions), `HiddenLivePill.tsx`, `ConnectionViewer.tsx`, `types.ts`. Connections is no longer a route; it's a tab inside `OrgRightPanel` (body in `_components/ConnectionsListBody.tsx`).
- `src/app/org/[githubLogin]/_components/` — view bodies and panel pieces: `OrgShell`, `OrgRail`, `OrgCanvasView`, `OrgRightPanel`, `NodeDetail` (Details tab body), `ConnectionsListBody` (Connections tab body), `WorkspacesView`, `MembersView`, `GraphView`, `useOrgView`. The non-canvas routes (`/initiatives`, `/workspaces`, `/members`, `/schematic`, `/graph`) are thin wrappers that mount the matching view component.
- `src/components/initiatives/` — shared `InitiativeDialog` / `MilestoneDialog` reused by the table UI and the canvas `+` menu.
- `src/lib/constants/prompt.ts` — `getCanvasPromptSuffix()` teaches the agent.

## Gotchas

- **Live vs authored.** Node ids prefixed `ws:` / `repo:` / `initiative:` / `milestone:` / `feature:` / `task:` are projected from the DB — never authored. The registry flags those categories `agentWritable: false` so they stay out of the tool schema. (`feature` is `userCreatable: true` since users can create features from the canvas `+` menu, but the agent still can't author them — see the "Feature creation" bullet below.) `task:` is registered as a live id prefix but tasks no longer project on any org canvas — task progress rolls into the parent feature's footer.
- **No milestone sub-canvas.** Milestones render as cards on their parent initiative's canvas alongside every feature anchored to that initiative. Milestone ↔ feature membership is shown by **projector-emitted synthetic edges** with id `synthetic:feature-milestone:<featureId>` (filtered out of the authored blob in `splitCanvas`). Clicking a milestone card is a no-op — it has no `ref`. The scope `milestone:<id>` is **not** a thing; legacy deep links carrying that ref fall through `parseScope` as `opaque` and project nothing. Feature ↔ milestone membership writes happen via three coordinated paths in `OrgCanvasBackground`: drawing a feature→milestone edge (intercepted in `handleEdgeAdd` → PATCH `Feature.milestoneId`), deleting a synthetic edge (intercepted in `handleEdgeDelete` → PATCH `milestoneId: null`), and dragging a feature card onto a milestone card (the existing Pairing 1 in `handleNodeDrop`). All three round-trip through `notifyFeatureReassignmentRefresh`.
- **DB-creating `+` menu.** `initiative`, `milestone`, and `feature` are visible in the user's `+` menu (`userCreatable: true`) but creation is **intercepted** in `OrgCanvasBackground`: it opens the matching dialog instead of dropping a node. On save → POST to API → Pusher refresh → projector re-emits the node. Click position is saved to `Canvas.data.positions[liveId]` so the new node lands where the user clicked.
- **Scope-aware `+` menu.** `categoryAllowedOnScope(id, ref)` in `canvas-categories.ts` is the single rule for which categories appear in the menu, and the same rule guards `handleNodeAdd`'s dispatch. Today: `initiative` only on root; `milestone` only on `initiative:<id>`; `feature` on every scope; `workspace`/`repository` never (`userCreatable: false`).
- **Feature creation — "most specific place wins".** `+` Feature on any canvas opens `CreateFeatureCanvasDialog`. The dialog resolves a `(workspaceId, initiativeId?)` pair per scope (root → user picks both; workspace canvas → workspace locked, no initiative; initiative canvas → initiative locked, user picks workspace). New features are **never created with a `milestoneId`** from this dialog — milestone membership is established afterward by drawing an edge from the feature card to a milestone card on the initiative canvas (intercepted as a `Feature.milestoneId` PATCH in `handleEdgeAdd`). The new feature renders on the **most specific** canvas matching the pair: initiative if set, else workspace — never on root, never on a milestone (no such scope). Right-click "Promote to Feature" on a `note` opens the same dialog with the note's text seeded as title/description and deletes the source note on save. **Edge-aware workspace picker:** when the dialog has an initiative anchor, it reads root canvas edges (`ws:<x> ↔ initiative:<y>`) and floats those workspaces to the top of the dropdown, preselecting the first linked one. Loose features (no initiative/milestone) project from `workspaceProjector`; every initiative-anchored feature (with or without a milestone) projects from `milestoneTimelineProjector`. The service-level `createFeature` validates that `milestone.initiativeId === initiativeId` (re-derives if omitted) and that the initiative belongs to the workspace's org — the projectors trust this invariant.
- **Side-channel DB writes from canvas interactions.** Most live-node fields are read-only: the splitter discards `text` / `category` / `customData` on live ids on autosave. Position (`x`/`y`) and size (`width`/`height`) are the exceptions — they ride through `Canvas.data.positions[liveId]` as a per-canvas overlay (size is optional; only set when the user has resized). Two other exceptions intercept in `handleNodeUpdate` and PATCH the REST API directly: the **milestone status swatch toolbar** (`milestone:` ids on the initiative canvas → PATCH `/api/.../milestones/[id]` with `{ status }`) and **feature card rename** (`feature:` ids, any canvas → PATCH `/api/features/[id]` with `{ title }`). Both follow the same shape: optimistic local update via `applyMutation` first, then fire-and-forget PATCH; the API route emits CANVAS_UPDATED via the matching `notifyFeature*Refresh` helper, which round-trips through the projector to reconcile the canonical value. The feature-rename path uses `notifyFeatureContentRefresh` (covers root + workspace canvas for loose features + parent initiative canvas for anchored features) — distinct from `notifyFeatureReassignmentRefresh`, which is for anchor moves and needs a `before` snapshot. New "edit a DB field from the canvas" features should follow this pattern (intercept → PATCH → matching content-refresh helper).
- **Group drags commit through `handleNodesUpdate`, NOT `handleNodeUpdate`.** The library fires `onNodesUpdate(updates[], canvasRef)` once per drag-end with every moved node — for a single-node drag that's a one-entry array, for a group drag it's the group plus every spatially-contained child. This is the only correct path because `applyMutation` reads `rootRef.current` / `subCanvasesRef.current`, both mirrored from React state via `useEffect` (so the mirror lags state by one commit). A synchronous loop of per-node `onNodeUpdate` calls all read the same stale starting state and each `setRoot` overwrites the previous one — visible bug: dragging a group leaves every child snapping back except the last one in the iteration order. `handleNodesUpdate` folds every patch through one `applyMutation` call so the group commits atomically. Don't fold the milestone-status special case in there — drag never carries `customData`, only x/y; toolbar status changes still come through `handleNodeUpdate`. Anyone wiring a new state mirror that drag commits read should keep this in mind, or add a synchronous-write path for the new state.
- **Drag-and-drop assignments.** The library exposes `canDropNodeOn(sources, target)` (predicate) and `onNodeDrop(sources, target, ctx)` (release handler) for node-on-node drops. When the predicate accepts, the library snaps the source back to its pre-drag position and fires `onNodeDrop` instead of committing a position update — so the consumer's job is purely to translate the drop into a DB or canvas-blob mutation. Three pairings live in `OrgCanvasBackground.handleNodeDrop` today, dispatched off `(source.category, target.category)`:
  1. **`feature:` → `milestone:` (DB reassign).** PATCH `/api/features/[id]` with `milestoneId`. The route's snapshot-before-update + `notifyFeatureReassignmentRefresh` fan out `CANVAS_UPDATED` on root and both initiatives the feature touched. The projector re-emits the feature on the parent initiative canvas with a fresh synthetic edge to the new milestone. Both the feature card AND the target milestone card live on the same initiative canvas, so this drop happens **without leaving the canvas the user is on**.
  2. **Authored callout (`note` / `decision` / `text`) → live container (`workspace:` / `initiative:`) (canvas move).** Removes the authored node from the source canvas blob (via `applyMutation`, so it's optimistic and Ctrl-Z-undoable) and read-modify-write PUTs it onto the target's sub-canvas blob (via direct `fetch`, so an unloaded target canvas still works). The node's id / text / category / customData / position survive verbatim — the user can drag it once they drill in. **Milestones are NOT a valid drop target** here (excluded from `LIVE_CONTAINER_CATEGORIES`) — they have no sub-canvas to land authored notes on. Lookup arrays `AUTHORED_DROPPABLE_CATEGORIES` and `LIVE_CONTAINER_CATEGORIES` live at module scope so the `useCallback` deps stay stable; both stay in lock-step with the registry in `canvas-categories.ts` and the `ref: liveId` projections in `lib/canvas/projectors.ts`.
  3. **`research:` → `initiative:` (DB reassign).** PATCH `/api/orgs/[githubLogin]/research/[researchId]` with `{ initiativeId }`. Structurally a sibling of pairing 1 (DB column update, no blob touch), but the source and target live on different canvases (root → initiative sub-canvas, or initiative A → initiative B). The route validates the target initiative belongs to the same org, snapshots the pre-write `initiativeId`, and `notifyResearchReassignmentRefresh` fans out `CANVAS_UPDATED` on both source and target refs. **Drop coords are intentionally not preserved** — the source's `(x, y)` has no meaning on a different canvas, and the projector's default initiative-canvas slot is the right landing spot. Symmetric "unscope to root" via drop isn't supported (root has no container card to drop onto); the API route accepts `initiativeId: null` so a future menu / "drop on empty canvas" gesture can wire it without a route change.
  Per-entity pusher modules live as siblings of `src/lib/canvas/pusher.ts`: **`feature-pusher.ts`** (`notifyFeatureCanvasRefresh` / `notifyFeatureContentRefresh` / `notifyFeatureReassignmentRefresh` — the canonical fan-outs for feature/task mutations across root + workspace + initiative canvases) and **`research-pusher.ts`** (`notifyResearchEvent` / `notifyResearchEventByLogin` for the viewer-side `RESEARCH_UPDATED` event, plus `notifyResearchReassignmentRefresh` for the canvas fan-out). All are re-exported from `@/lib/canvas`. **The convention:** a live-node category that needs more than a single `notifyCanvasUpdatedByLogin` call gets its own `<entity>-pusher.ts` next to `pusher.ts` — the entity owns the ref-resolution logic, the canvas owns the wire format. Wholesale agent / REST / webhook callers all import via the barrel; the helpers themselves use relative `./pusher` imports so the dependency direction stays explicit (canvas-internal → canvas-internal).
  Predicate must be cheap (id-prefix sniff + category check, no fetches) — it runs every drag-frame. Future pairings (task → feature, feature → feature dependency, note → live-node-as-comment) layer in additional clauses on the same predicate without changing the library shape. The snap-back default is intentional: the most common follow-up moves the source to a different sub-canvas, so leaving it at drop coords for one frame would be visually meaningless and could write a `positions[liveId]` overlay for a node that no longer projects on the source canvas.
- **Edge gestures as DB writes (feature ↔ milestone).** A user-drawn edge between a feature card and a milestone card on the initiative canvas is intercepted in `handleEdgeAdd`: `detectFeatureMilestoneEdge` sniffs the endpoint id prefixes, the edge is **suppressed from the authored blob** (an `applyMutation(removeEdge)` undoes the library's optimistic write), and `patchFeatureMilestone(featureId, milestoneId)` PATCHes `Feature.milestoneId`. The Pusher fan-out + projector re-run produce the corresponding `synthetic:feature-milestone:<featureId>` edge on the next read — visually identical from the user's POV. Symmetric on delete: `handleEdgeDelete` sniffs the `synthetic:feature-milestone:` prefix and PATCHes `milestoneId: null`. `handleEdgeUpdate` short-circuits on synthetic ids (the blob has no edge with that id; the DB-projected version isn't editable through `customData`). This pattern — gesture intercepted, DB mutated, synthetic edge appears via projection — is the canonical way to express any DB-backed relationship as a canvas edge. Authored edges (feature→feature dependencies, note→workspace links) keep flowing through the default blob path.
- **`currentRef` state** tracks which canvas the user is looking at (driven by `SystemCanvas`'s `onNavigate` and `onBreadcrumbClick`). It drives the HiddenLivePill's per-scope refetch, the scope-aware `+` menu, and the URL sync.
- **Deep links via `?canvas=<ref>`.** Mount-only: on first paint the URL's `?canvas=` is read synchronously into a ref, a spinner overlay covers the canvas, and `SystemCanvas`'s imperative `zoomIntoNode(ref, { durationMs: 0 })` drills in once root has loaded. The library only navigates from the currently-rendered canvas, so we always mount root first; the overlay hides the brief flash. After mount the URL is **write-only** — `handleBreadcrumbsChange` is the single source of truth for `currentRef` and just mirrors the deepest scope into `?canvas=` for refresh / share. The right panel viewers use separate query params: `?c=<slug>` opens a connection in the Connections tab; `?r=<slug>` opens a research doc in the Details tab (see "Research deep links" below). All four (`?canvas=`, `?c=`, `?r=`, `?chat=`) coexist on the same URL because each is owned by a different consumer in `OrgCanvasView`. **There is intentionally NO runtime-reactive `?canvas=` follower.** We tried one (so the AttentionList could drill in by writing `?canvas=&select=`) and it kept fighting the lib's own navigation: `router.replace` lags `useSearchParams` by a render, so on every drill-in the effect saw `urlCanvasRef === ""` (stale) vs `currentRef === "initiative:..."` (fresh) and on every breadcrumb-back it saw the inverse — either case fired `zoomIntoNode` and bounced the user back. Both directions of the asymmetry kept popping up under different interactions. The simpler thing is to not have a follower at all: AttentionList opens its target page in a new tab (see below).
- **Research deep links via `?r=<slug>`.** Reactive (unlike `?canvas=`): an effect in `OrgCanvasView` watches `searchParams.get("r")` and on change resolves the slug → research id by fetching `/api/orgs/[githubLogin]/research`, then synthesizes a `selectedNode` of the form `{ id: "research:<id>", text: topic, category: "research", customData: { slug } }` and sets it. `OrgRightPanel` auto-flips to the Details tab on `selectedNode?.id` change, `NodeDetail` recognizes the `research:` prefix and short-circuits to `<ResearchViewer>` (same path a real canvas click takes). Stale slug = strip the param so refresh doesn't loop. **The canvas itself does not visually highlight the deep-linked node** — the lib owns selection state via its own click events, and we don't have a "select this node" imperative handle today; the UX priority is "show the writeup," not "draw a halo on the card." This is the canonical pattern for "agent emits a chat link → opens an in-page viewer": the agent writes a relative markdown link `[anchor](?r=<slug>)` (taught in `getCanvasPromptSuffix`), `SidebarChatMessage` intercepts the click via a custom `ReactMarkdown` `a` component (`IN_PAGE_PARAMS = {"r","c"}`) and routes through `router.replace` so chat / canvas / streaming state survives, the URL effect picks up the new slug, and the viewer mounts. Adding a new agent-link surface (e.g. a future `?n=<liveId>` for "select this node") = add a watcher effect alongside the `?r=` one, add the param to `IN_PAGE_PARAMS`, teach the prompt to emit it. The link interceptor falls back to `target="_blank"` for any href that isn't a recognized in-page param, so absolute URLs from research citations open in a new tab without nuking canvas state.
- **AttentionList opens external.** The synthetic intro card (`_components/AttentionList.tsx`) used to drill the canvas in-place via `?canvas=&select=`, but that's the path that fought `useSearchParams` lag (see above). It now `window.open`s `item.link` (`/w/<slug>/plan/<id>` or `/w/<slug>/task/<id>`) in a new tab — the user can act on the item in the workspace context, and their org-canvas state is untouched.
- **Inline task chat (`<TaskChat>`).** Sibling to `<FeaturePlanChat>`, mounted in `NodeDetail.tsx` `case "task":`. Reads from `/api/tasks/[id]/messages`, posts to `/api/chat/message` (the same endpoint the full task page uses) with `taskId` set, subscribes to the task's Pusher channel for live updates. Mirrors `<FeaturePlanChat>`'s structural choices (compact scroll container, optimistic-then-Pusher message pattern, IN_PROGRESS+pending-form input gating); only the API endpoints and the artifact-rendering policy differ. The bubble fork lives in `<TaskChatMessage>`.
- **Task artifact rendering policy** (`<TaskChatMessage>` + `<TaskArtifactPill>`):
  1. **`FORM`** → inline interactive (`FormArtifact` reused from the full task page). User clicks an option → optimistic POST to `/api/chat/message` with `replyId` set → planner branch fires.
  2. **`PULL_REQUEST`** → inline render (`PullRequestArtifact` reused). PR status is a primary signal in the canvas's "what did the agent do" loop and earns inline space.
  3. **`CODE` / `DIFF` / `LONGFORM` / `BOUNTY` / `PUBLISH_WORKFLOW`** → small clickable pill (`TaskArtifactPill`) → click opens a large `<Dialog>` mounting the matching artifact panel from `src/app/w/[slug]/task/[...taskParams]/artifacts/`. These panels are render-only (no workspace/pod context required), so they slot cleanly into the modal.
  4. **`BROWSER` / `IDE` / `WORKFLOW` / `GRAPH` / `MEDIA` / `BUG_REPORT` / `STREAM` / `TASKS` / `VERIFY` / non-clarifying `PLAN`** → pill with an external-link glyph, click opens `/w/<slug>/task/<id>` in a new tab. These artifacts depend on rich workspace context (pod URLs, `useWorkspace`, multi-tab state, etc.) that the modal can't faithfully reproduce — making the artifact's existence *visible* without faking an inline render.
  Adding a new artifact-type renderer = extend `KIND_META` in `TaskArtifactPill.tsx` (with the right `mode`) plus a case in `renderModalBody` for `inModal` types. Don't bring `BROWSER`/`WORKFLOW`/`GRAPH` into `inModal` without a plan for their context-prop surface.
- **HiddenLivePill is per-canvas, not root-only.** `Canvas.data.hidden` is a per-ref overlay — every canvas (root, workspace sub-canvas, initiative sub-canvas) has its own hidden list. `hiddenLive` tracks the **current** canvas's list and is refetched whenever `currentRef` changes; the pill auto-hides when empty so sub-canvases with nothing hidden cost zero chrome. The chat workspace seed is **separate**: `rootHiddenLive` always tracks the root list and is the only thing piped through `onHiddenChange` — drilling into a sub-canvas must not re-seed the chat with repo/milestone entries.
- **Both `hiddenLive` and `rootHiddenLive` are `HiddenLiveEntry[] | null` (null = not yet fetched), NOT `[]`.** The `onHiddenChange` notify effect short-circuits while `rootHiddenLive` is null. `OrgCanvasView` gates the chat mount on `hiddenInitialized` (set by the first non-stub `onHiddenChange` callback) so `defaultExtraWorkspaceSlugs` is seeded correctly; firing with an empty stub would seed the chat with no filtering and the real list would land too late (DashboardChat reads the prop only on mount).
- **Right panel is resizable — give every panel an explicit `id` and `order`.** `OrgCanvasView` wraps the canvas + sidebar in a `<ResizablePanelGroup autoSaveId="org-right-panel">` so the user's chosen sidebar width persists across reloads. The library (`react-resizable-panels`) keys the saved layout in localStorage by panel `id`; if you don't pass an explicit `id`, it falls back to a `useId()` value that changes on every fresh mount, the saved layout fails to match, and the lib throws "No group element found for id" from a pointer handler — which crashes the whole canvas (the throw is caught by the nearest error boundary, but there isn't one between this and the page root). Same rule for the group element itself. Keep `id="org-right-panel-group"` on the group, `id="org-right-panel-filler"` / `order={1}` on the transparent left filler, and `id="org-right-panel-sidebar"` / `order={2}` on the visible sidebar. Adding a third panel = give it an `id` and an `order` — never rely on render order alone when `autoSaveId` is set. Sizing constants (`SIDEBAR_DEFAULT_SIZE` / `SIDEBAR_MIN_SIZE` / `SIDEBAR_MAX_SIZE` / `SIDEBAR_EXPANDED_SIZE`, all percent of container width) live at the top of `OrgCanvasView` — `MAX_SIZE` is generous (80) because the sidebar hosts the full `<ConnectionViewer>` (see next bullet).
- **Connection viewer renders inside the sidebar, not over the canvas.** Clicking a connection in the Connections tab does *not* take over the page — `<ConnectionViewer>` mounts inside the Connections tab body in `OrgRightPanel`, swapping with the list (the list stays mounted via `hidden` so flipping back is instant). On open, `OrgCanvasView.handleConnectionClick` imperatively grows the sidebar to `SIDEBAR_EXPANDED_SIZE` via the panel ref (only if currently smaller) and snapshots the prior size in `preExpandSizeRef`. `handleBack` restores the prior size, but only if the panel is still parked at `SIDEBAR_EXPANDED_SIZE` — if the user manually resized while viewing, we respect their new width. The opening flow also auto-flips the active tab to "connections" via an effect keyed on `activeConnection?.id`. Deep links via `?c=<slug>` route through the same `handleConnectionClick`, so a fresh load lands directly in the expanded-sidebar viewer. **Don't add a second mechanism that resizes the sidebar imperatively** — pair every imperative `panel.resize()` with the same snapshot/restore protocol or the user's chosen layout will get clobbered.
- **Edges link to Connection docs via `edge.customData.connectionId`.** A canvas edge can carry a free-form string `connectionId` in its `customData`; the Connections tab uses it to render the doc that "lives between" the edge's endpoints. The id round-trips verbatim through the splitter (`src/lib/canvas/io.ts` passes edges through untouched), so no schema change. The library's `CanvasEdge` type doesn't include `customData` — we cast through `(edge as { customData?: ... }).customData` in four places (`OrgCanvasView.readEdgeConnectionId`, `ConnectionsListBody.readEdgeConnectionId`, `OrgRightPanel.edgeLinksToConnection`, plus `compactEdge` / `toCanvasEdge` in `src/lib/ai/canvasTools.ts` which surface the field to the agent on `read_canvas` and accept it on `update_canvas`/`patch_canvas`); centralizing in a shared helper isn't worth the import surface for one accessor. Orphaned ids (the connection was deleted) self-clean: the link mode treats unresolvable ids as "not linked" and the user can re-pick on the next click. **The agent's path is the canonical reason `customData` survives `compactEdge`** — without it, `read_canvas` output would strip `connectionId` and the agent couldn't discover the slug↔edge linkage from canvas inspection, leaving `read_connection(slug)` unreachable through normal flow.
- **Edge click → Connections tab in link mode.** Selection state flows through the lib's `onSelectionChange` callback (added in `system-canvas-react@0.1.13`) — a single source of truth that fires atomically on every state transition: node-click, edge-click, **canvas-background click (deselect)**, Escape, Delete, navigation, and stale-selection collapse. `OrgCanvasBackground` translates the lib payload into `SelectionWithLabels` (its own type) by enriching the edge case with human-readable `fromLabel` / `toLabel` resolved via `getNodeLabel(node)` from the canvas's own node list — live nodes carry the entity's real name in `text`, so a workspace edge endpoint surfaces as e.g. `"Mock Org Frontend"`, not `ws:cmoti…`. `OrgCanvasView.handleSelectionChange` is the one consumer-side handler: it sets `selectedNode` / `selectedEdge` (always exactly one or zero), opens the viewer when an edge has a linked connection, and closes an edge-owned viewer on deselect. **Don't reach for `onNodeClick` / `onEdgeClick` to track selection** — those fire on activation only and miss every deselect path; `onSelectionChange` is the right primitive. When `selectedEdge` becomes non-null, `OrgRightPanel` auto-flips to the Connections tab. **Link-mode chrome is intentionally sparse**: no header strip, no global "edge selected" banner — just a single `+ New connection` button at the top of the list and a small link icon next to every connection row. The chrome is gated on `selectedEdge !== null && customData.connectionId === null` (i.e. only when the edge is selected AND has no link yet); a linked edge bypasses the list entirely and opens its viewer directly. Selection has no visible × cancel — the natural deselect paths (clicking a different edge, clicking a node, **clicking the canvas background**, navigating, hitting Back from the viewer) cover it. The viewer's optional `onUnlink` prop is wired only when the open connection's id matches the selected edge's link, so list-driven opens render Back-only. **Back from a linked-edge viewer also clears `selectedEdge`** — without that clear, the user would land in a list with no link-mode chrome (because the rule hides `+`/link icons on linked edges), which reads as "selected but invisibly so."
- **Imperative edge patches via `edgePatchHandleRef`.** Linking and unlinking write `customData.connectionId` through the canvas's existing `applyMutation` → `updateEdge` path. To avoid prop-drilling a callback through every Connections list row, `OrgCanvasBackground` exposes its `handleEdgeUpdate` through an optional `MutableRefObject` prop (`edgePatchHandleRef`); the parent (`OrgCanvasView`) declares the ref and passes it down. The canvas's effect writes the latest closure into `ref.current` on every render and nulls it on unmount. The link write also locally patches `selectedEdge.edge.customData` so a follow-up unlink reads the fresh state. This pattern is the right escape hatch when "the canvas owns the data, but a sibling triggers the write" — don't reach for state lifting; the canvas blob is large and lifting it would re-render every consumer on every drag.
- **Chat input prefill via `pendingInputDraft`.** The link-mode `+ Create connection` button doesn't dispatch any DB write or call any agent tool — it composes a chat message *for* the user (`Make a connection document between A and B`), writes it to `useCanvasChatStore.pendingInputDraft`, and the panel switches tabs to Chat. `<SidebarChatInput>` watches the slot in an effect, applies the value to its local `input` state, focuses + caret-to-end, then immediately clears the slot via `setPendingInputDraft(null)` so a tab-switch back doesn't re-apply. The user can edit and send normally. This is the canonical pattern for "canvas affordance composes a chat message" — proposals, agent suggestions, etc. should reuse `pendingInputDraft` rather than reaching imperatively into the textarea ref.
- **Right panel is tabbed, not routed.** `OrgRightPanel` owns Chat / Details / Connections tabs. Tab state is panel-local — switching tabs leaves the canvas selection intact, so the user can flip back and forth without losing which node they were inspecting. **Chat is the default landing tab.** The Details tab auto-activates when the user clicks a node (effect keyed on `selectedNode?.id`, so reselecting the same node still re-fires); manual tab clicks override until the next selection change. The chat tab is gated on a `chatReady` flag (workspaces loaded + hidden list initialized + `?chat=<shareId>` preload resolved + active conversation started in the store) so it never mounts with a half-loaded context. **All three tab bodies stay mounted** (toggled with the HTML `hidden` attribute, not unmounted) so scroll position, input focus, in-flight streaming, and Pusher subscriptions survive tab switches. Live nodes (id prefixed `ws:` / `repo:` / `initiative:` / `milestone:` / `feature:` / `task:`) fetch detail from `/api/orgs/[githubLogin]/canvas/node/[liveId]`; authored nodes (`note`, `decision`) render `node.text` directly. Adding a new live kind = teach `NodeDetail`'s `KindExtras` switch and the node-detail API route.
- **Canvas chat state lives in `useCanvasChatStore` (`_state/canvasChatStore.ts`), not in `SidebarChat` local state.** The store exists because several upcoming features — LLM-generated `propose-canvas-change` artifacts (rendered as both a chat card AND a halo on the affected canvas node), sub-agents (kicked off from chat, surfaced on the nodes they're working on), live task-status / PR-list artifacts — create bidirectional flow between the chat and the canvas itself. Two `useState` islands talking via prop drilling would be miserable. The store is the single source of truth; chat and canvas both subscribe with narrow selectors. **Conversation lifecycle is owned by `OrgCanvasView`, not `SidebarChat`** — `OrgCanvasView` calls `startConversation` once `chatReady` is true (workspaces + hidden list + share preload all resolved), then keeps the conversation's `context` (canvas ref, breadcrumb, selected node id) in sync via `updateActiveContext`. `SidebarChat` is a pure renderer over the active conversation; it never owns chat state, so unmounting it (e.g. a tab switch) is cheap and idempotent. **Auto-save runs at the page level**, not inside `SidebarChat` — `useCanvasChatAutoSave` mounts in `OrgCanvasView` and survives chat unmounts, so pending writes never get dropped by tab navigation.
- **Store performance contract — read this before adding a consumer.** The streaming write path commits one `set()` per stream chunk (~50/sec). To keep that cheap:
  1. **Always select.** Never call `useCanvasChatStore()` without a selector — that subscribes to the whole store and re-renders on every text-delta. Use `useCanvasChatStore((s) => s.x)` instead. The single biggest footgun.
  2. **Use `useShallow` for derived collections.** A selector that returns a fresh array/object on every call (e.g. `Array.from(s.conversations.values())` or `messages.map(...)`) defeats Zustand's `Object.is` bail-out → re-renders constantly. Wrap with `useShallow` from `zustand/react/shallow`.
  3. **Stable empty-array sentinels.** When a selector might return `undefined` (e.g. no active conversation), fall back to a module-level `const EMPTY_X: T[] = []` instead of `?? []` — the latter creates a new `[]` every time, breaking identity equality. See `EMPTY_MESSAGES` / `EMPTY_TOOL_CALLS` in `SidebarChat.tsx`.
  4. **Select narrowly.** `SidebarChat` selects `messages`, `isLoading`, `activeToolCalls` separately rather than the whole conversation object, so the canvas's `proposals` selector is unaffected by chat streaming, and vice versa. When the canvas grows badge/halo subscribers for proposals/sub-agents/artifacts, follow the same pattern: select the specific slice you render, not "the conversation."
- **Adding a canvas-bound chat feature?** Use the reserved slots in `useCanvasChatStore`. Proposals → `state.proposals` map; long-running agent runs → `state.subAgentRuns`; rich artifacts → `state.artifacts`. Each is keyed by id; messages reference them via `message.artifactIds`. The chat renders the inline card from `state.artifacts[id]`; the canvas reads the same id from the same map and renders its badge. If you find yourself emitting a Pusher event from the chat to the canvas (or vice versa), stop — that's what the store is for.
- **Synthetic intro / "top items needing your attention".** First registered artifact type is `attention-list` (`src/app/org/[githubLogin]/_components/AttentionList.tsx`). On fresh canvas entry `OrgCanvasView` fetches `/api/orgs/[githubLogin]/attention?limit=3` (backed by `src/services/attention/topItems.ts`, which fans 4 signal queries — halted/failed workflows, features awaiting user feedback, tasks with FORM artifact, ready-to-review COMPLETED — across the user's accessible workspaces in the org), then seeds `startConversation` with one synthetic assistant message + a registered artifact of `type: "attention-list"`. The seed is **ephemeral**: `startConversation` accepts `ephemeralSeedCount`, which `useCanvasChatAutoSave` reads to skip leading messages from the first POST/PUT — so the synthetic intro never lands in `chat_conversations` and never leaks into `?chat=<shareId>` shares. Skip conditions: `?chat=<shareId>` is present (forking), or `sessionStorage["hive:attention-dismissed:<login>"] === "1"` (user clicked × this session). Dismissal flows through `state.dismissedArtifactIds` (a generic per-session dismiss set the dispatcher reads) — `OrgCanvasView` subscribes once and writes the sessionStorage flag whenever an `attention-*` id lands in it. **`MessageArtifacts` in `SidebarChat` is the single dispatcher** — switch on `artifact.type` and add a render arm. Per-card behavior (callbacks, etc.) goes through store actions, not function-valued artifact data, so the artifact registry stays serializable.
- **What's NOT in this store.** Voice transcript / mic state lives in `src/stores/useVoiceStore.ts` (separate concern, separate lifetime — mic-session-scoped, available across surfaces). When voice transcription wants to drop a finalized transcript into the canvas chat, it calls `useCanvasChatStore.getState().appendUserMessage(...)`. **The dashboard chat does NOT use this store.** `DashboardChat` keeps its own local state on purpose; the two surfaces are diverging. Don't try to unify them — when a feature lives on both, share the lower-level pieces (`useStreamProcessor`, message types), not the state layer.
- **Adding a category** = add to `canvas-categories.ts` (with `agentWritable` / `userCreatable` flags) + add a matching `CategoryDefinition` in `canvas-theme.ts` (same `id`; load-time check throws on mismatch).
- **Adding a projected entity** = new `Projector` in `src/lib/canvas/projectors.ts` + add the id prefix to `LIVE_ID_PREFIXES` in `src/lib/canvas/scope.ts`. Card width / row layout go in `geometry.ts` so theme and projector see the same numbers.
- **Cap unbounded projections.** Loose features on workspace / initiative sub-canvases are capped to `LOOSE_FEATURE_LIMIT` (25) ordered by `updatedAt desc`. Without the cap, deep backlogs (hundreds or thousands of features) project as a single giant horizontal row and the library's auto-fit zooms out until cards are unreadable. Any new projector that emits a per-row layout from a potentially large table needs the same treatment — pick a `*_LIMIT` constant, sort by `updatedAt desc`, and trust the long tail to a list view elsewhere. User-positioned overrides via `Canvas.data.positions[liveId]` are unaffected by the cap.
- **Adding decorative bands (columns / rows / lanes).** A projector returns optional `columns?` / `rows?` on its `ProjectionResult`. `readCanvas` merges them onto the returned `CanvasData`; the library renders them as background bands + headers. Bands are decorative — cards never snap to them. See `buildTimelineColumns` in `projectors.ts` for the milestone-timeline pattern.
- **Pusher invalidation.** Mutations to Initiatives/Milestones must emit `CANVAS_UPDATED` on the org channel with the right `ref` (root for initiatives; `initiative:<id>` for milestones; both for milestone changes that affect the root-level rollup). Use `notifyCanvasUpdatedByLogin` / `notifyCanvasesUpdatedByLogin` from `@/lib/canvas`.
- **Research nodes — first authored→live category, swap-by-text-dedupe.** Research is the canvas's first node type with a two-phase lifecycle: the user picks `+ Research` (allowed on root or initiative scope per `categoryAllowedOnScope`), drops an authored `research` node, types a topic, and on the first text edit per id `OrgCanvasBackground.handleNodeUpdate` fires a synthetic `Research: <topic>` chat message via `useSendCanvasChatMessage` (no dialog, no DB write yet, no client-side bookkeeping beyond a `firedResearchKickoffsRef` Set to prevent double-fires per node id — the library prefills new nodes with placeholder text like `"New node"` so the trigger is "first user-driven text edit," not "empty→non-empty"). The agent's `save_research` tool (`src/lib/ai/researchTools.ts`) creates the row with `topic` set to the user's verbatim wording (the prompt suffix demands this), Pusher fires `CANVAS_UPDATED`, the projector emits a live `research:<id>` node carrying the same text. **The authored→live "swap" is NOT a client-side concern** — `dedupeAuthoredResearch` in `src/lib/canvas/io.ts` runs on every read AND every write, dropping any authored `research` node whose trimmed text matches a live `research:<id>` node's text and carrying the authored node's `(x, y)` into `positions[liveId]` if no overlay exists. Read-side dedupe makes the user see one node on the next refresh; write-side dedupe makes the blob converge to clean state on the next autosave (self-healing). Then `update_research` lands the markdown content; the projector flips `customData.status` from `"researching"` to `"ready"` (derived from `Research.content !== null`); the spinner badge in the `topRightOuter` slot disappears; the right-panel `ResearchViewer` (`_components/ResearchViewer.tsx`) streams the content in via the `RESEARCH_UPDATED` Pusher event. **The chat is the source of truth for research lifecycle** (every save/update is a tool call on a `CanvasChatMessage`), but the canvas node is the primary surface — chat just records what happened. The agent can also call `save_research` unprompted mid-conversation (no authored placeholder to dedupe against; the new live node lands at the projector's default position, fine). **Why dedupe at the IO boundary instead of an in-flight client-side swap.** An earlier version tracked pending kickoffs in a `pendingResearchSwapsRef` queue and removed authored nodes via `applyMutation(removeNode)` when `save_research` tool outputs landed in the chat-store stream. It never converged: autosave (600ms debounce) raced the in-flight Pusher refetch from `save_research` (300ms server-side delay) which raced the position-overlay PUT, and any combination losing the race left the authored node alive in the canvas blob — a phantom that re-appeared on every subsequent refetch. The IO-boundary dedupe sidesteps every race by being purely a derived-data filter: read-side hides on render, write-side cleans on persist, no ordering required. Adding a new authored→live category that follows this pattern = mirror `dedupeAuthoredResearch` for the new pair (e.g. `dedupeAuthoredQuestion` if questions get a similar `+ Question` flow), and call it from `readCanvas` + `splitCanvas` next to the existing call.
