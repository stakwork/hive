# Org Canvas

A live whiteboard on the org page. Initiatives and Milestones are real DB rows projected onto the canvas; humans create them through the `+` menu (which opens a dialog and hits the REST API). The agent annotates around them.

**Design doc:** `docs/plans/org-initiatives.md` — read it before non-trivial changes. (Old plan `docs/plans/org-canvas.md` is superseded.)

## The short version

- `src/lib/canvas/` — projection pipeline. `readCanvas(orgId, ref)` merges DB-projected live nodes (workspaces, repos, initiatives, milestones) with the authored `Canvas.data` blob. `writeCanvas` splits merged docs back into an authored-only blob. `geometry.ts` is the shared source of truth for card sizes / row layout. `pusher.ts` is the shared `CANVAS_UPDATED` emitter.
- `src/lib/ai/canvasTools.ts` — agent tools (`read_canvas`, `update_canvas`, `patch_canvas`), each with optional `ref`. Agent does NOT create initiatives or milestones — those are human-only via the `+` menu.
- `src/app/api/orgs/[githubLogin]/canvas/*` — REST routes (root + `[ref]` + `hide` + `node/[liveId]`), all thin wrappers over `@/lib/canvas`. The `node/[liveId]` route is the single org-scoped read for live-node detail (used by the right panel's Details tab).
- `src/app/api/orgs/[githubLogin]/initiatives/*` — REST routes for the DB-backed Initiative + Milestone models. Every mutating call emits `CANVAS_UPDATED` so open canvases refetch.
- `src/app/org/[githubLogin]/layout.tsx` + `_components/OrgShell.tsx` — server layout does auth + org lookup once; `OrgShell` renders the left icon rail (`OrgRail`) and either full-bleed children (canvas / schematic / graph) or a centered max-w container with org header (initiatives / workspaces / members). `useOrgView` maps pathname → view id and `viewIsFullBleed` decides which layout mode.
- `src/app/org/[githubLogin]/page.tsx` + `_components/OrgCanvasView.tsx` — default route. `OrgCanvasView` is the two-layer composition: full-bleed canvas (`OrgCanvasBackground`) on the left, fixed-width right panel (`OrgRightPanel`) on the right with three tabs — **Chat** (`SidebarChat`, the default landing tab; `_components/SidebarChat.tsx`), **Details**, and **Connections**. The canvas chat is also forkably shareable: the Share button in the chat header copies a `/org/<login>?chat=<shareId>` URL; landing on that URL preloads the conversation into a fresh continuable chat (each forker's own auto-save row).
- `src/app/org/[githubLogin]/_state/` — Zustand store + side-effect hooks for the canvas chat. `canvasChatStore.ts` is the source of truth for conversations, messages, streaming state, and reserved slots (`proposals`, `subAgentRuns`, `artifacts`) that future canvas-bound features will fill. `useSendCanvasChatMessage.ts` owns the streaming send path (fetch → `useStreamProcessor` → store writes). `useCanvasChatAutoSave.ts` is the write-through subscription that persists messages to `chat_conversations` via the workspace endpoint. The store is **org-canvas-scoped on purpose** — the dashboard chat keeps its own local state and does NOT use this store.
- `src/app/org/[githubLogin]/connections/` — canvas internals: `OrgCanvasBackground.tsx` (the canvas UI itself), `canvas-categories.ts` (category registry + scope rules — single source of truth), `canvas-theme.ts` (renderer definitions), `HiddenLivePill.tsx`, `ConnectionViewer.tsx`, `types.ts`. Connections is no longer a route; it's a tab inside `OrgRightPanel` (body in `_components/ConnectionsListBody.tsx`).
- `src/app/org/[githubLogin]/_components/` — view bodies and panel pieces: `OrgShell`, `OrgRail`, `OrgCanvasView`, `OrgRightPanel`, `NodeDetail` (Details tab body), `ConnectionsListBody` (Connections tab body), `WorkspacesView`, `MembersView`, `GraphView`, `useOrgView`. The non-canvas routes (`/initiatives`, `/workspaces`, `/members`, `/schematic`, `/graph`) are thin wrappers that mount the matching view component.
- `src/components/initiatives/` — shared `InitiativeDialog` / `MilestoneDialog` reused by the table UI and the canvas `+` menu.
- `src/lib/constants/prompt.ts` — `getCanvasPromptSuffix()` teaches the agent.

## Gotchas

- **Live vs authored.** Node ids prefixed `ws:` / `repo:` / `initiative:` / `milestone:` are projected from the DB — never authored. The registry flags those categories `agentWritable: false` so they stay out of the tool schema.
- **DB-creating `+` menu.** `initiative` and `milestone` are visible in the user's `+` menu (`userCreatable: true`) but creation is **intercepted** in `OrgCanvasBackground`: it opens the matching dialog instead of dropping a node. On save → POST to API → Pusher refresh → projector re-emits the node. Click position is saved to `Canvas.data.positions[liveId]` so the new node lands where the user clicked.
- **Scope-aware `+` menu.** `categoryAllowedOnScope(id, ref)` in `canvas-categories.ts` is the single rule for which categories appear in the menu, and the same rule guards `handleNodeAdd`'s dispatch. Today: `initiative` only on root; `milestone` only on `initiative:<id>`; `workspace`/`repository` never (`userCreatable: false`).
- **Side-channel DB writes from canvas interactions.** Most live-node fields are read-only: the splitter discards `text` / `category` / `customData` on live ids on autosave. Position (`x`/`y`) and size (`width`/`height`) are the exceptions — they ride through `Canvas.data.positions[liveId]` as a per-canvas overlay (size is optional; only set when the user has resized). The other exception is the **milestone status swatch toolbar**, which intercepts in `handleNodeUpdate` and PATCHes the milestone REST endpoint directly. Optimistic local update + Pusher reconcile. New "edit a DB field from the canvas" features should follow that PATCH pattern.
- **`currentRef` state** tracks which canvas the user is looking at (driven by `SystemCanvas`'s `onNavigate` and `onBreadcrumbClick`). It drives the HiddenLivePill's per-scope refetch, the scope-aware `+` menu, and the URL sync.
- **Deep links via `?canvas=<ref>`.** On mount, the URL's `?canvas=` is read synchronously into a ref, a spinner overlay covers the canvas, and `SystemCanvas`'s imperative `zoomIntoNode(ref, { durationMs: 0 })` drills in once root has loaded. The library only navigates from the currently-rendered canvas, so we always mount root first; the overlay hides the brief flash. The connection-doc viewer uses a separate `?c=<slug>` param (managed by `OrgCanvasView`) so the two deep links coexist.
- **HiddenLivePill is per-canvas, not root-only.** `Canvas.data.hidden` is a per-ref overlay — every canvas (root, workspace sub-canvas, initiative sub-canvas) has its own hidden list. `hiddenLive` tracks the **current** canvas's list and is refetched whenever `currentRef` changes; the pill auto-hides when empty so sub-canvases with nothing hidden cost zero chrome. The chat workspace seed is **separate**: `rootHiddenLive` always tracks the root list and is the only thing piped through `onHiddenChange` — drilling into a sub-canvas must not re-seed the chat with repo/milestone entries.
- **Both `hiddenLive` and `rootHiddenLive` are `HiddenLiveEntry[] | null` (null = not yet fetched), NOT `[]`.** The `onHiddenChange` notify effect short-circuits while `rootHiddenLive` is null. `OrgCanvasView` gates the chat mount on `hiddenInitialized` (set by the first non-stub `onHiddenChange` callback) so `defaultExtraWorkspaceSlugs` is seeded correctly; firing with an empty stub would seed the chat with no filtering and the real list would land too late (DashboardChat reads the prop only on mount).
- **Right panel is tabbed, not routed.** `OrgRightPanel` owns Chat / Details / Connections tabs. Tab state is panel-local — switching tabs leaves the canvas selection intact, so the user can flip back and forth without losing which node they were inspecting. **Chat is the default landing tab.** The Details tab auto-activates when the user clicks a node (effect keyed on `selectedNode?.id`, so reselecting the same node still re-fires); manual tab clicks override until the next selection change. The chat tab is gated on a `chatReady` flag (workspaces loaded + hidden list initialized + `?chat=<shareId>` preload resolved + active conversation started in the store) so it never mounts with a half-loaded context. **All three tab bodies stay mounted** (toggled with the HTML `hidden` attribute, not unmounted) so scroll position, input focus, in-flight streaming, and Pusher subscriptions survive tab switches. Live nodes (id prefixed `ws:` / `repo:` / `initiative:` / `milestone:` / `feature:` / `task:`) fetch detail from `/api/orgs/[githubLogin]/canvas/node/[liveId]`; authored nodes (`note`, `decision`) render `node.text` directly. Adding a new live kind = teach `NodeDetail`'s `KindExtras` switch and the node-detail API route.
- **Canvas chat state lives in `useCanvasChatStore` (`_state/canvasChatStore.ts`), not in `SidebarChat` local state.** The store exists because several upcoming features — LLM-generated `propose-canvas-change` artifacts (rendered as both a chat card AND a halo on the affected canvas node), sub-agents (kicked off from chat, surfaced on the nodes they're working on), live task-status / PR-list artifacts — create bidirectional flow between the chat and the canvas itself. Two `useState` islands talking via prop drilling would be miserable. The store is the single source of truth; chat and canvas both subscribe with narrow selectors. **Conversation lifecycle is owned by `OrgCanvasView`, not `SidebarChat`** — `OrgCanvasView` calls `startConversation` once `chatReady` is true (workspaces + hidden list + share preload all resolved), then keeps the conversation's `context` (canvas ref, breadcrumb, selected node id) in sync via `updateActiveContext`. `SidebarChat` is a pure renderer over the active conversation; it never owns chat state, so unmounting it (e.g. a tab switch) is cheap and idempotent. **Auto-save runs at the page level**, not inside `SidebarChat` — `useCanvasChatAutoSave` mounts in `OrgCanvasView` and survives chat unmounts, so pending writes never get dropped by tab navigation.
- **Store performance contract — read this before adding a consumer.** The streaming write path commits one `set()` per stream chunk (~50/sec). To keep that cheap:
  1. **Always select.** Never call `useCanvasChatStore()` without a selector — that subscribes to the whole store and re-renders on every text-delta. Use `useCanvasChatStore((s) => s.x)` instead. The single biggest footgun.
  2. **Use `useShallow` for derived collections.** A selector that returns a fresh array/object on every call (e.g. `Array.from(s.conversations.values())` or `messages.map(...)`) defeats Zustand's `Object.is` bail-out → re-renders constantly. Wrap with `useShallow` from `zustand/react/shallow`.
  3. **Stable empty-array sentinels.** When a selector might return `undefined` (e.g. no active conversation), fall back to a module-level `const EMPTY_X: T[] = []` instead of `?? []` — the latter creates a new `[]` every time, breaking identity equality. See `EMPTY_MESSAGES` / `EMPTY_TOOL_CALLS` in `SidebarChat.tsx`.
  4. **Select narrowly.** `SidebarChat` selects `messages`, `isLoading`, `activeToolCalls` separately rather than the whole conversation object, so the canvas's `proposals` selector is unaffected by chat streaming, and vice versa. When the canvas grows badge/halo subscribers for proposals/sub-agents/artifacts, follow the same pattern: select the specific slice you render, not "the conversation."
- **Adding a canvas-bound chat feature?** Use the reserved slots in `useCanvasChatStore`. Proposals → `state.proposals` map; long-running agent runs → `state.subAgentRuns`; rich artifacts → `state.artifacts`. Each is keyed by id; messages reference them via `message.artifactIds`. The chat renders the inline card from `state.artifacts[id]`; the canvas reads the same id from the same map and renders its badge. If you find yourself emitting a Pusher event from the chat to the canvas (or vice versa), stop — that's what the store is for.
- **What's NOT in this store.** Voice transcript / mic state lives in `src/stores/useVoiceStore.ts` (separate concern, separate lifetime — mic-session-scoped, available across surfaces). When voice transcription wants to drop a finalized transcript into the canvas chat, it calls `useCanvasChatStore.getState().appendUserMessage(...)`. **The dashboard chat does NOT use this store.** `DashboardChat` keeps its own local state on purpose; the two surfaces are diverging. Don't try to unify them — when a feature lives on both, share the lower-level pieces (`useStreamProcessor`, message types), not the state layer.
- **Adding a category** = add to `canvas-categories.ts` (with `agentWritable` / `userCreatable` flags) + add a matching `CategoryDefinition` in `canvas-theme.ts` (same `id`; load-time check throws on mismatch).
- **Adding a projected entity** = new `Projector` in `src/lib/canvas/projectors.ts` + add the id prefix to `LIVE_ID_PREFIXES` in `src/lib/canvas/scope.ts`. Card width / row layout go in `geometry.ts` so theme and projector see the same numbers.
- **Adding decorative bands (columns / rows / lanes).** A projector returns optional `columns?` / `rows?` on its `ProjectionResult`. `readCanvas` merges them onto the returned `CanvasData`; the library renders them as background bands + headers. Bands are decorative — cards never snap to them. See `buildTimelineColumns` in `projectors.ts` for the milestone-timeline pattern.
- **Pusher invalidation.** Mutations to Initiatives/Milestones must emit `CANVAS_UPDATED` on the org channel with the right `ref` (root for initiatives; `initiative:<id>` for milestones; both for milestone changes that affect the root-level rollup). Use `notifyCanvasUpdatedByLogin` / `notifyCanvasesUpdatedByLogin` from `@/lib/canvas`.
